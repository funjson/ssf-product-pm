# ssf-product-pm 交接上下文

交接日期：2026-05-25  
适用对象：在另一台电脑、另一个 Codex 会话中继续维护 `ssf-product-pm` 的 Agent。

## 1. 当前目标

本仓库正在建设一个产品经理阶段的 AI Skill：`ssf-product-pm`。

它服务于更大的 `super-spec-flow / ssf` 研发流程体系，定位是：

> 生成 AI 可消费的产品分析与产品设计文档，支撑后续架构师 AI、研发 AI、测试 AI、原型 AI 和前端 AI。

当前关注点不是做传统 PRD，也不是做项目管理系统，而是把产品分析和产品设计做成：

- 有流程的。
- 有阶段门禁的。
- 有稳定 ID 的。
- 可追踪的。
- 可被后续 AI 角色消费的。
- 能在真实企业研发场景里反复运行的。

## 2. 重要设计共识

当前 skill 的核心流程分为两个大阶段：

```text
Analysis 分析阶段
  - 信息收集
  - 市场/竞品/客户现场/用户需求调研
  - 需求分析
  - analysis-human-review 人工评审

Design 设计阶段
  - 产品架构设计
  - product-architecture-human-review 人工评审
  - PRD
  - 功能任务规格
  - UI 信息架构
  - 结构化 UI / 交互规格
  - 原型 Prompt 与 UI 标注
  - 产品基线与变更说明
```

关键规则：

- Analysis 结束必须中断，等待人工评审。
- 产品架构结束必须中断，等待人工评审。
- PRD、功能任务、UI、原型标注、基线使用自动自检查。
- Design 必须先做产品架构，再做后续文档。
- 新项目和迭代功能都需要考虑产品架构，只是影响范围不同。
- 不能每次小改都全量重跑，应通过 `change-run` 判断影响范围。
- 写文件前必须读取 `ssf-workspace/index.md` 和目标实例 `manifest.md`。
- 不能把新事项覆盖到旧实例。
- 功能任务中的每个 `FEAT` 必须同构展开。
- UI 中的每个 `SCR` 必须同构展开。
- 缺失字段不要省略，应写“无 / 暂无 / 待确认”。

## 4. 不要重复踩的坑

### 4.1 不要把模板当流程规则

模板只描述文档结构。  
阶段规则、小阶段规则、评审规则应放在 `flows/`、`core/`、`registries/`、`references/` 中。

### 4.2 不要跳过人工评审

Analysis 和产品架构都是强交互节点。  
它们不是自动自检查，必须中断等待用户确认。

### 4.3 不要覆盖旧实例

用户可能同时做多个完全不同的产品事项。  
每次落盘前必须先看 `ssf-workspace/index.md` 判断目标实例。

如果用户说“重新生成”“再来一版”“换个方向”“做另一个事情”，必须先执行 intake gate，不能直接覆盖旧目录。

### 4.4 不要让模型压缩 FEAT 或 SCR

之前多次验证发现，模型容易把多个功能任务或多个 UI 页面压缩成总表。  
当前规则要求：

- 每个 `FEAT` 必须完整展开。
- 每个 `SCR` 必须完整展开。
- 输出过长时分批生成，不得写“同上”或“略”。

### 4.5 不要在 PM 阶段写技术实现

PM 阶段可以写产品模块、业务对象、输入输出、业务规则、验收标准。  
不要写数据库表、接口路径、缓存方案、消息队列、部署方案等技术实现细节。

## 6. Figma Prompt 的设计方向

Figma prompt 不应该直接吃所有文档原文。更合理的方式是从当前实例的 `product-spec/` 中抽取以下输入：

优先级从高到低：

1. `04-product-architecture.md`
   - 产品模块。
   - 模块边界。
   - `REQ -> MOD -> FEAT` 映射。
2. `07-ui-ia-screen-inventory.md`
   - 页面清单。
   - `SCR`。
   - 导航关系。
3. `08-structured-ui-interaction-spec.md`
   - 页面布局。
   - `CMP`。
   - 交互。
   - 状态。
   - 权限。
4. `09-prototype-prompt-ui-annotation.md`
   - 原型生成要求。
   - UI 标注要求。
5. `06-feature-task-spec.md`
   - `FEAT`。
   - 输入输出。
   - 主流程。
   - 分支流程。
   - 业务规则 `BR`。
   - 验收标准 `AC`。
6. `05-prd.md`
   - 产品定位。
   - 目标用户。
   - 范围边界。
7. `10-product-baseline-change.md`
   - 如果是变更版本，用来确认当前基线和影响范围。

`01-03` 分析阶段文档通常不适合完整塞给 Figma，只应抽取背景、目标用户、关键场景。

Figma prompt 应强约束：

- 每个 `SCR` 一个 Frame。
- 每个关键组件保留 `CMP`。
- 不新增规格外页面、功能、按钮和业务流程。
- 原型是可评审原型，不是生产前端代码。
- 保留 `REQ / MOD / FEAT / SCR / CMP / BR / AC` 追踪关系。
- 必须覆盖默认态、加载态、空态、错误态、禁用态、无权限态。

## 9. 下一步建议

建议继续按这个顺序推进：

1. 不改仓库，先设计 Figma prompt 方案。
2. 用已有或新生成的 `product-spec/04-10` 组装一次 Figma prompt。
3. 在 Figma 中生成原型。
4. 观察 Figma 是否能保留页面、组件、状态和 ID。
5. 根据验证结果，再决定是否把 prompt 资产纳入 skill。
6. 如果要纳入，先设计目录和注册表，不要临时打补丁。

核心判断标准：

- Figma 能否按 `SCR` 生成页面。
- Figma 能否按 `CMP` 组织组件。
- Figma 能否表达关键交互和状态。
- Figma 输出是否能回流到 `09-prototype-prompt-ui-annotation.md` 或后续 UI 标注。
- 前端 AI 能否基于“产品规格 + Figma 原型 + UI 标注”继续实现，而不是直接复制 Figma 代码。
