# ssf-product-pm 工作流说明

## 1. 设计目标

该工作流用于把传统产品经理的产品设计活动改造成 AI 可消费的产品规格产物。

传统 PM 依赖会议、口头解释、原型图和经验补全；AI 研发链路不能依赖这些隐性上下文。因此本工作流要求输出结构化、可追踪、可测试的文档。

## 2. 默认阶段

```text
产品调研与洞察
  ↓
需求分析
  ↓
PRD 产品总上下文
  ↓
功能任务规格
  ↓
UI 信息架构
  ↓
结构化 UI/交互规格
  ↓
原型生成 Prompt 与 UI 标注
  ↓
产品基线与变更说明
```

## 3. 各阶段说明

### 3.1 产品调研与洞察

合并项目基本描述、用户现场/访谈调研、市场调研、竞品分析。目标不是写厚报告，而是形成“需求为什么成立”的证据链，并产生会影响产品设计的洞察。

### 3.2 需求分析

解释调研洞察、用户现场需求和业务问题如何转化为需求。需求分析先于 PRD，用来定义需求项、需求边界和待确认问题。

### 3.3 PRD

基于调研和需求分析，提供产品背景、目标、范围、角色、约束和产品级成功标准。

### 3.4 功能任务规格

核心执行文档。每个功能任务都包含目标、用户故事、输入输出、主流程、分支流程、异常流程、业务规则、权限、关联页面、验收标准、测试关注点。

### 3.5 UI 信息架构

定义页面树、页面清单、导航关系、页面状态和页面与功能的映射。

### 3.6 结构化 UI/交互规格

定义页面布局、组件、交互、校验、状态、权限表现和 UI 测试断言。

### 3.7 原型生成 Prompt 与 UI 标注

支持 AI 原型工具生成原型，并将原型元素和需求、功能、规则、验收标准绑定。

### 3.8 产品基线与变更说明

记录当前版本、已确认功能、页面、规则、关键决策、废弃设计和变更影响范围。

## 4. 不做的事情

- 不做传统需求池管理。
- 不做优先级矩阵。
- 不做 Roadmap 或下一期规划。
- 不基于人力资源不足拆 MVP。
- 不在 PM 阶段写数据库表、接口路径、缓存和消息队列等技术设计。

## 5. 生成策略

- 如果用户要求生成或修改文件，必须先读取 `ssf-workspace/index.md` 判断目标实例；不存在时先创建工作区。
- 如果用户请求与现有实例明显不是同一件事，必须新建实例或请求确认，不得覆盖。
- 如果用户要求完整产品设计包，按 8 个模板生成。
- 如果用户只要求某个阶段，只生成该阶段，并标记依赖的上游输入和缺失输入。
- 如果信息不足，优先在文档中写“假设与待确认问题”；只有实例归属不清、覆盖风险或关键事实缺失时才提问。
- 如果涉及已有系统变更，必须先读取实例 `manifest.md` 和产品基线，再生成变更说明。

## 6. 实例与跳阶段规则

### 6.1 实例判断

每个独立产品、功能事项或变更任务都是一个实例。实例由 `ssf-workspace/index.md` 统一登记，由 `instances/<instance-id>/manifest.md` 记录状态。

判断为新实例的典型情况：

- 用户明确表示新建、另一个产品、不要覆盖旧内容。
- 产品名称、业务领域、目标用户、核心目标与最近实例明显不同。
- 当前请求无法解释为旧实例的变更。

判断为旧实例续跑的典型情况：

- 用户明确表示继续、基于之前、补齐上次。
- 用户指定了实例 ID、产品名或已有文档路径。
- 请求内容与已有基线一致，只是补充某个阶段或文档。

### 6.2 跳阶段执行

允许用户直接执行某个阶段，但必须显式说明：

- 本阶段需要哪些上游输入。
- 当前实例已有哪些上游输入。
- 缺失哪些上游输入。
- 本次基于哪些假设生成。
- 哪些内容需要后续回填到上游文档。
