# 质量检查清单

## 1. 通用检查

| 检查项 | 通过标准 |
|---|---|
| 是否解析目标实例 | 生成或修改文件前读取 `ssf-workspace/index.md`，明确目标实例 |
| 是否执行 Intake Gate | 重新生成、覆盖、跳阶段、变更或信息不清时先执行 intake |
| 是否避免覆盖 | 新事项与旧实例不一致时新建实例或请求确认 |
| 是否更新实例状态 | 生成或修改文件后更新 `index.md` 和实例 `manifest.md` |
| 是否读取工作流配置 | 执行前读取 workflow、action-commands、template-index、review-gates |
| 是否有文档目标 | 每个文档开头说明用途和下游消费者 |
| 是否有版本信息 | 每个文档包含 document_id、version、generated_at |
| 是否使用统一 ID | 关键对象都有稳定 ID |
| 是否有待确认问题 | 信息不足处明确标注，不把假设当事实 |
| 是否避免技术越界 | PM 文档不提前写死架构实现 |
| 是否正确设置状态 | 未经用户确认不得写 approved / confirmed |

## 2. 分析输入检查

| 检查项 | 通过标准 |
|---|---|
| 是否有产品基本信息 | 产品名称、形态、领域、目标客户、真实用户、核心目标清楚或标记待确认 |
| 是否记录信息来源 | 每项输入区分用户提供、模型假设、待确认 |
| 是否有客户现场入口 | 客户访谈、现场观察、客服销售反馈、旧系统资料至少有一类，缺失则标记 |
| 是否有缺失信息处理方式 | 明确先提问、模型假设、只生成提纲或中断 |

## 3. 调研文档检查

| 检查项 | 通过标准 |
|---|---|
| 是否有项目基本描述 | 产品形态、业务领域、目标用户、当前现状、设计目标和约束清楚 |
| 是否合并市场、用户、竞品 | 三类输入在同一文档中表达 |
| 是否强化用户调研 | 有客户现场、访谈、客服销售反馈或旧系统资料等来源记录 |
| 是否区分事实和分析 | 用户原话/观察事实与产品经理分析分开表达 |
| 是否输出洞察 | 每条调研结论必须形成洞察 |
| 洞察是否影响设计 | 每条洞察说明对需求、UI、流程或规则的影响 |
| 是否剔除无关内容 | 不进入设计的内容明确列出 |

## 4. 需求分析检查

| 检查项 | 通过标准 |
|---|---|
| 是否属于分析阶段 | 需求分析基于分析输入、调研洞察、用户需求和业务问题生成 |
| 是否有用户需求归纳 | 原始需求、真实问题、关联角色和场景明确 |
| 是否说明需求边界 | 每个核心 REQ 有包含内容、不包含内容和边界原因 |
| 是否映射后续产物 | REQ 能映射到建议 MOD、GOAL 和 FEAT |
| 是否触发人工评审 | 分析阶段结束后必须进入 analysis-human-review |

## 5. 产品架构检查

| 检查项 | 通过标准 |
|---|---|
| 是否是设计第一节点 | 进入设计阶段后先生成或更新产品架构 |
| 是否有产品模块 | 核心需求被归纳到 MOD 模块 |
| 是否有模块边界 | 每个 MOD 有职责、包含内容、不包含内容 |
| 是否有业务对象 | 核心业务对象用 OBJ 表达，但不写数据库表 |
| 是否有映射关系 | REQ -> MOD -> FEAT 追踪完整 |
| 是否有 Mermaid 图 | 至少包含产品模块图和用户任务流/模块协作图两个 Mermaid 代码块 |
| 是否有架构决策选项 | 说明模块划分策略，并给出用户可确认/修改的决策项 |
| 是否触发人工评审 | 产品架构结束后必须进入 product-architecture-human-review |
| 是否避免技术越界 | 不写接口、数据库、缓存、队列、部署 |

## 6. PRD 检查

| 检查项 | 通过标准 |
|---|---|
| 是否基于产品架构 | PRD 引用 PRODUCT-ARCH-001，不直接从需求跳到功能 |
| 是否提供产品总上下文 | 背景、定位、目标、范围、角色、约束清楚 |
| 是否不过度细化 | 细节不挤进 PRD，而进入功能任务规格 |
| 是否有产品级成功标准 | 成功标准可验证但不替代功能验收 |

## 7. 功能任务规格检查

| 检查项 | 通过标准 |
|---|---|
| 是否按功能任务组织 | 每个 FEAT 独立描述 |
| 是否结构同构 | 每个 FEAT 都保留完整模板小节，空内容写“无 / 暂无 / 待确认” |
| 是否拒绝压缩 | 不出现“其他同上”“FEAT-x 至 FEAT-y 汇总” |
| 是否关联模块 | 每个 FEAT 关联 MOD 或明确标记“模块待补齐” |
| 是否说明模块边界 | 每个 FEAT 有“模块归属与边界”小节 |
| 是否包含流程 | 主流程、分支流程、异常流程齐全 |
| 是否包含规则 | 业务规则结构化，有 BR ID |
| 是否包含验收标准 | 每个核心功能有 AC，且可测试 |
| 是否关联 UI | 功能关联 screen_id 和 component_id |
| 是否给测试输入 | 有测试关注点，不只写功能说明 |
| 是否自动自检查 | 执行 feature-spec-auto-review，失败进入 repair-run |

## 8. UI 文档检查

| 检查项 | 通过标准 |
|---|---|
| 页面是否来自功能 | 每个页面关联至少一个 FEAT |
| 页面是否关联模块 | 每个 SCR 关联至少一个 MOD |
| 页面是否结构同构 | 每个 SCR 都保留完整页面模板小节 |
| 是否拒绝页面压缩 | 不出现“SCR-x 至 SCR-y 关键组件”这类合并总表 |
| 组件是否可追踪 | 关键组件有 CMP ID |
| 是否覆盖状态 | 默认、加载、空态、错误、权限按需覆盖 |
| 是否包含交互 | 用户动作、系统响应和跳转清楚 |
| 是否包含标注 | 原型元素能映射到需求、规则、验收 |
| 是否自动自检查 | 执行对应 UI review gate，失败进入 repair-run |

## 9. 变更文档检查

| 检查项 | 通过标准 |
|---|---|
| 是否有基线版本 | 明确 base_version 和 target_version |
| 是否写变更前 | 说明旧功能、旧页面、旧规则、旧验收 |
| 是否写变更后 | 明确新行为、新页面、新规则、新验收 |
| 是否有影响范围 | 标记受影响的功能、页面、组件、规则、验收 |
| 是否有回归关注点 | 给测试 AI 明确回归范围 |
