# ID 规范

## 目标

统一所有产品设计文档中的对象 ID，保证分析输入、调研、需求分析、产品架构、PRD、功能任务、UI、验收、变更之间可以互相追踪。

## 命名建议

- 对外展示可以使用中文名称。
- 文档内部追踪必须使用 ID。
- 同一对象不要在不同文档中创建多个 ID。
- `SPI-xxx` 代表一个独立产品设计实例，不能把无关产品混入同一个实例。
- 用户调研链路建议保持 `SRC -> RAW -> INS -> NEED -> REQ`，避免把用户原话直接写成功能。
- 产品架构链路建议保持 `REQ -> MOD -> FEAT`，避免需求直接平铺成功能任务。
- 功能任务是主执行维度，但必须归属到产品模块或明确标记“模块待补齐”。
- 自动检查项必须带 Review Gate 来源前缀，不使用裸编号 `CHECK-001`。例如 PRD 使用 `CHECK-PRD-001`，功能任务使用 `CHECK-FEAT-001`，变更使用 `CHECK-CHG-002-001`。

## 示例链路

```text
SRC-001 客户现场访谈
  → RAW-001 用户说“我每天都要先看核心数据是否异常”
  → INS-001 用户希望快速查看核心数据
  → NEED-001 需要在首页快速识别关键指标状态
  → GOAL-001 提升核心数据可见性
  → REQ-001 需要首页核心指标展示能力
  → MOD-001 健康概览模块
  → FEAT-001 首页核心指标展示
  → SCR-001 首页
  → CMP-001 核心指标卡片
  → BR-001 指标异常状态展示规则
  → AC-001 首页核心指标展示验收标准
```
